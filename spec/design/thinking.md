## 1. 非文本元素的"流式揭示"(element-aware streaming)

### 核心洞察

大模型最终吐出的是 markdown。**文本**逐字到达 → 逐字平滑揭示(smoother + spawn 淡入)是对的。但**非文本结构**(表格 / 列表 / 公式 / 图片 / 链接)有**内部结构**,逐字淡入或"等完整再一次性全刷"都不理想——它们应该按**自身结构的节奏**揭示,而且**流式过程中会 reflow**(宽度/行数/布局随后续 token 变化)。

> "聊天过程中,非文字的 streaming 效果"是值得专门设计的一层,而不是套用文本的揭示。

### 按元素拆(理想揭示节奏 + 流式中的问题)

- **表格 table**
  - 理想:**先出 header 行 → 列结构/网格浮现 → 单元格里再填字**。
  - 流式问题:① markdown 表格在**分隔行 `|---|` 到达前不可判定**(看着就是带 `|` 的文本)→ 何时"确认是表格";② 列宽由最宽单元格决定,**全部行流完才知道** → 列宽**渐进变化**(reflow,顶开/挤压前面已渲染的格)。
- **列表 list**
  - 理想:**逐项**出(marker 先现 → 项内文字填);嵌套深度确认后缩进。
  - 流式问题:行首 `-`/`1.` 在确认前**有歧义**(可能是普通文本);有序号/嵌套深度会变。
- **公式 formula(LaTeX)**
  - 理想:**不逐字显示原文 TeX**(`$x^` 半截无效、且原文闪现→排版替换很糟)→ 用**占位/骨架** hold,直到可闭合再**一次性 typeset 出现**。
  - 流式问题:宽高在完整前未知 → 占位尺寸估计;依赖 0013(C 纹理 / B RaTeX)。
- **图片 image**
  - 理想:`![alt](url)` 的语法**不闪现成文本**;`)` 到达 → 占位框(估高/已知宽高比)→ 纹理 swap(embed FSM,0007)。
  - 流式问题:真实尺寸到达时 reflow。
- **链接 link**
  - 理想:`[text](url)` 的可见文字正常流;`](url)` 部分**不闪现**;闭合确认后把文字**"提升"成链接样式**(上色/下划线)。
  - 流式问题:`[text` 在 `](` 到达前有歧义。

### 横切设计问题

- **A. 歧义 / 确认延迟**:很多结构要等后面的分隔符才能判定(表格要分隔行、链接要 `](url)`、围栏要 ```` ``` ````、公式要 `$`)。**"立刻显示" vs "别显示会被替换的原始语法"** 是核心张力。已有工具:**hold 区(0006)** + **remend(0004 §5.1)**。需要的是**按元素的"揭示策略"**:文本=立即流;结构=hold 到可判定再揭示/提升。
- **B. 揭示编排(每类一套)**:表格(header→网格→格内字)、列表(逐项)、图片(占位→纹理)、公式(骨架→typeset)各有自然顺序 → 给每类定一套 choreography。
- **C. 流式中的 reflow = 动画,不是 pop**:后续 token 让结构变大/列变宽时,已渲染部分应**平滑移动到新位**(= §2「字块 move」),而非消失重排。**这把"非文本流式"和"动画式 reflow"两个命题接上了。**
- **D. 占位 / 骨架尺寸**:不可提前渲染的(公式/图片)用什么占位 + 估计尺寸,避免真身到达时跳变。
- **E. "提升"转场**:文本→表格、文本→链接、原文→typeset 公式——确认那一刻应是**平滑提升**,不是闪一下重建。
- **F. 与块冻结的关系**:这些揭示/reflow 只发生在**正在生长的尾部块**;settled 后冻结(0002/Plan2 G)。

### 与已有机制 / 想法的接点

- hold 区(0006)解"是不是某结构"的歧义;remend(0004 §5.1)解"已确认但没写完";二者是 A 的基础。
- embed FSM(0007 Placeholder→Loading→Ready)是图片/公式占位的现成骨架。
- 动画式 reflow(TODO2 ★)是 C/E 的实现机制(逐字 GPU 实例 + `target_pos` lerp + 稳定 keying)。
- 公式渲染走 0013(v1 纹理 embed / 升级 RaTeX)。

### 待办(想清楚后上提)

- [ ] 定义"**元素揭示策略**"接口:每种非文本类型 → { 何时确认、占位形态、揭示编排、reflow 动画 }。
- [ ] 表格的"渐进列宽"算法 + 动画(最典型的 reflow)。
- [ ] 公式/图片的占位尺寸估计。
- [ ] 与 hold 区 / remend / embed FSM 的统一(它们其实都是"揭示策略"的特例)。
- 成型后 → 上提为 ADR(如"0014 非文本流式揭示")或并入相关相位(P 标签层 / O 嵌入)。

## 2. 字块 move(动画式 reflow / FLIP / magic-move)

### 问题

流式 markdown/LaTeX 里,后续字符让**已渲染的前序字符位置变化**:`$x` → `$x^2$` 把 x 抬成上标、表格列变宽顶开前面、`*` → `**bold` 重排。当前是"**消失 → 回退 → 重写**"(闪)。想要的是:旧字形**平滑移动到新位置**。

### 这算什么问题

**带对象恒等性的动画式布局过渡**(keyed reflow / **FLIP** / "magic move")。核心是字形**身份稳定**——知道"这个字从 A 移到 B",而非"旧的没了、新的出现"。

### 业界实现

- 通用 UI:D3 data-join + key(object constancy)、**FLIP**(Paul Lewis)、Framer Motion `layout`、SwiftUI `matchedGeometryEffect`、Flutter Hero、Keynote Magic Move。
- 数学专用、最贴近:**Manim `TransformMatchingTex`**(匹配两版 LaTeX 的子式,动画移动/变形)。
- **流式 markdown 里几乎没人做** → 潜在差异化点。

### 我们的架构正合适

已有逐字 GPU 实例 + `spawn_time`(淡入)→ 加 `target_pos` + `move_start`,着色器在两位置间 lerp(与淡入同机制,0011 §3.2)。所以**渲染端代价很小**,难点不在画。

### 真正的难点 = 稳定 keying

要按"**内容偏移 / 逻辑身份**"匹配旧↔新字形(**不是**按屏幕位置),才能知道谁移到哪。即一个 **keyed diff**(可借 warp 的行级 diff / Manim 的子式匹配思路)。流式 append 时前缀稳定较易;reflow(列变宽、上下标)需逻辑身份匹配。

### 依赖

数学要做这个**必须 RaTeX 把数学字形喂进 quad 管线(0013 B)**;opaque 纹理(0013 C)里的字形无法 tween。

### 与 §1 的关系

这是 §1「非文本流式」里 reflow(C)/ 提升(E)的**实现机制**——两个命题同源:非文本结构在流式中变形,就靠"字块 move"平滑过渡,而非重建。

### 开放问题

- keying 策略:按内容偏移?cluster 身份?插入怎么不打乱前缀匹配?
- 动画曲线 / 时长;大量字同时 move 的性能(用 compute 批量更新 `target_pos`?)。
- 与块冻结:只在尾部块做;settled 块不动。
